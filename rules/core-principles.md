# 中核となる原則

## シンプルかつミニマム

- **シンプル**: 複雑なアーキテクチャよりも、理解しやすくメンテナンスしやすい単純な構造を優先する。
- **ミニマム**: コード量、ファイル数、依存ライブラリを必要最小限に抑える。
- **使うかもは不要**: 今必要な機能だけを実装する。将来必要になるかもしれない機能は、必要になった時に追加する。
- **不要になったら即削除**: 不要、冗長な記述・ファイルは発見次第、確実に削除する。一応残しておきたい場合は Git で残し、必要な時に復元する。

## 明示的な情報整理

- **明示的な記述**: 人間が読んでも AI が読んでも同一のオペレーションができる記述を心がける。
- **記載場所を明確に**: 進行状況や検討事項など、記載場所が明確に定めてある場合は、その場所に記載する。絶対に他の場所に記載しない。
- **信頼できる唯一の情報源**: 情報の重複を避け、常に一つの情報源を参照するようにする。

## 反復的な改善

- **小さく始める**: 小さく始め、積極的にフィードバックを得て、改善するサイクルをテンポよく回す。
- **早期に失敗する**: 問題があれば早期に発見し、修正する。早期発覚を妨げるエラーハンドリング等を避ける。

## 人間中心

- **承認プロセス**: 重要な意思決定はユーザーの承認のもと進める。特に修正に時間のかかる変更や影響範囲の大きい変更は必ず承認を得る。
- **ドキュメンテーション**: 常に最新の状態を保ち、後から参画した人間が簡単に大枠の把握と必要な作業を行えるようにする。
